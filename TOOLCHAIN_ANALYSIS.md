# Почему 1С++ (icpp) нельзя собрать новым компилятором. Анализ тулчейна

Документ фиксирует, ПОЧЕМУ проект не переносится на современные компиляторы
(VS2019/2022 и т.п.) и какой тулчейн нужен для рабочей `1cpp.dll`.
Основано на эксперименте в ветке `build/vs2022` (см. `BUILD_VS2022.md`).

## Главный факт: библиотеки 1С — это MFC42 (VC6)

`1cv7.exe` и его DLL (`BkEnd.dll`, `type32.dll`, `seven.dll`, …) собраны
**Visual C++ 6.0 и линкуют `MFC42.DLL`**. Import-либы в `Source/LIBS/*.lib`
экспортируют символы с искажением имён (name mangling) именно этой эпохи.

Прямой замер (`dumpbin /LINKERMEMBER` по bkend/type32/basic/seven/br32):

| Форма `CString` в экспортах либ | Кол-во символов |
|---|---|
| `class CString` (VC6/MFC42, манглинг `…VCString@@…`) | **906** |
| `CStringT<…>` (MFC 7.0+, VS2003 и новее) | **0** |

Вывод однозначен: **весь бинарный интерфейс 1С построен на `class CString`
(MFC42).** Любой код, который хочет вызывать API 1С, обязан использовать ТАКОЙ
ЖЕ `CString` — и по имени (для линковки), и по layout в памяти (для работы).

## Что сломалось в MFC начиная с VS2003 (MFC 7.0)

- **`CString` перестал быть классом.** Стало `typedef CStringT<char,
  StrTraitMFC_DLL<…>> CString;`. Следствия:
  - **Линковка**: вызов `Foo(CString&)` манглится как `…CStringT@…` и НЕ
    совпадает с экспортом либы `…VCString@@…` → «неразрешённый внешний символ».
    В нашем VS2022-эксперименте так отвалилось **45 из 52** символов.
  - **Рантайм**: у `CStringT` другой layout, чем у MFC42-`CString`. Даже если
    заставить слинковаться, объекты `CString`, которыми обменивается `1cpp.dll` и
    `1cv7.exe`, будут интерпретироваться по-разному → порча памяти, падения.
- **`CObject` стал абстрактным** (`virtual ~CObject() = 0;`) — в MFC42 не так.
- **Изменились message-map/RTTI-макросы** (`GetThisClass`, `GetThisMessageMap`,
  `GetMessageMap() const override`) — ещё 7 из 52 неразрешённых символов.

Эти изменения — часть публичного ABI MFC. Обойти их без замены всей строково-
объектной модели MFC на MFC42-совместимую реализацию нельзя. Практически это
означает: **новый компилятор с новым MFC для этого проекта непригоден.**

## Итог VS2022-эксперимента (ветка build/vs2022)

- Компиляция — **100%** (все 117 .cpp), после ~15 категорий правок под MFC140.
- Линковка — **невозможна**: 52 неразрешённых символа, все из-за ABI MFC42↔MFC140.
- Даже форсированная линковка (thunk-и на каждый символ) дала бы DLL, которая
  падает в 1С из-за несовпадения layout `CString`/`CObject`.

## Какой тулчейн нужен

| Вариант | Компиляция | Линковка с либами 1С | Работа в 1cv7 |
|---|---|---|---|
| **VC6 (MFC42)** | под вопросом (старый слабый компилятор для кода 2010 г.) | **ДА** — `class CString` совпадает с либами | **ДА** — ABI совпадает |
| Оригинал: **Intel C++ 10.1 + VS2005 + STLport + Boost** | ДА (это и есть build-конфиг проекта) | ДА — через shim `forwarder` (VC6-ремаппер, бриджит CString) | ДА | 
| VS2019/2022 (MFC 14x) | ДА | **НЕТ** (ABI) | НЕТ |

- **VC6** даёт правильный ABI «бесплатно» (родной `class CString` = MFC42), но
  его компилятор — 1998 года, со слабой поддержкой шаблонов; код v3.2.4.3 писался
  под VS2005/Intel, и не факт, что VC6 его переварит. Нужны также **STLport** и
  **Boost версии, совместимой с VC6** (Boost ≤ ~1.34).
- **Оригинальный тулчейн** (Intel C++ 10.1 + VS2005) гарантированно компилирует
  (это его родной build-конфиг) и линкуется через `forwarder`. Минус: Intel C++
  10.1 сейчас практически недоступен. Но у проекта есть и «VS»-конфигурации
  (`ReleaseVS`) — чистый компилятор VS2005 без Intel; если найдётся VS2005 +
  STLport + Boost, это самый предсказуемый путь.

## Рекомендация

Для гарантированно рабочей `1cpp.dll` целиться нужно в **MFC42-совместимый
тулчейн**. Порядок предпочтения:
1. **VC6 + STLport + Boost(старый)** — ABI идеален; риск в компиляции (проверяется).
2. **VS2005 (+ STLport + Boost) по «VS»-конфигурации** — родной build-конфиг
   проекта, компиляция гарантирована, ABI бриджится штатным `forwarder`.
3. Intel C++ 10.1 — только если найдётся дистрибутив (для полного соответствия
   оригиналу).

Современные компиляторы (VS2017+) исключены на уровне ABI — это не вопрос правок
исходников, а вопрос бинарной совместимости с «замороженными» MFC42-библиотеками
1С 7.7.
