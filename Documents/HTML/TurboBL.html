splitchild="0"> </chminfo> <div class="section" id="id4"> <h2><a name="id4">Для всех</a></h2> <p>Что происходит внутри 1С, когда мы пишем спр.ТекущийЭлемент()?</p> <p>1С преобразует имя метода ТекущийЭлемент в соответствующий ему номер, и вызывает метод объекта по номеру. НО, клинические исследования показали, что для ЛЮБОГО объекта 1С используется один алгоритм такого преобразования, реализованный "В ЛОБ", и котрый заключатся в простом последовательном переборе всех имен методов объекта и сравнении их с искомым. Соответственно, чем больше номер метода, тем большее время занимает такой поиск. Например, для того же "ТекущийЭлемент" перед реальным вызовом функции происходит перебор 88 названий методов.</p> <p>Аналогично и для свойств объектов.</p> </div> <div class="section" id="rainbow"> <h2><a name="rainbow">Для знакомых с "Rainbow"</a></h2> <p>Как известно, для поиска номера метода применяется</p> <pre class="literal-block"> virtual int CBLContext::FindMethod(const char* name); </pre> <p>Как оказалось, НИ В ОДНОМ из объектов 1С данный метод не переопределен. Перехват вызова функций различных объектов показал лень и мягко говоря, не вполне компетентность людей, писавших этот код.</p> <p>Вобщем, когда вызывается <tt class="docutils literal"><span class="pre">FindMethod</span></tt>, из него вызываются: <tt class="docutils literal"><span class="pre">GetNMethods(),</span> <span class="pre">GetMethodName(i,0),</span> <span class="pre">GetMethodName(i,1)</span></tt> т.е. <tt class="docutils literal"><span class="pre">FindMethod</span></tt> работает примерно так:</p> <pre class="literal-block"> for(int i=0;i<GetNMethods();i++) { if(!stricmp(name,GetMethodName(i,0)) return i; if(!stricmp(name,GetMethodName(i,1)) return i; } return -1; </pre> <p>Понятно, хотелось добиться универсальности и простоты, что и достигнуто. Но при этом: - Некомпетентность: GetNMethods можно вынести из цикла. - Лень: не зря же FindMethod виртуальная, почему же она не переопределена ни в одном из наследуемых классов для более оптимального поиска?</p> <p>Аналогичная ситуация и с FindProp.</p> </div> </div> <div class="section" id="id5"> <h1><a name="id5">Цель компоненты</a></h1> <chminfo splitchild="0"> </chminfo> <p>Дабы устранить досадную оплошность разработчиков 1С в отношении поиска методов и свойств была создана ВК <strong>TurboBL</strong>, которая включена в 1С++. Суть ее работы в следующем: оптимизация алгоритма преобразования имен методов/свойств в их номера.</p> <p>Попутно сделана возможность обращаться к методам и переменным модулей ГрупповыхКонтекстов.</p> <div class="section" id="id6"> <h2><a name="id6">Как использовать</a></h2> <pre class="literal-block"> ЗагрузитьВнешнююКомпоненту("1cpp.dll"); </pre> <p>Вот практически и все.</p> <p>Более никаких изменений в существующем коде конфигурации не требуется. При загрузке компонента берет на себя всю работу по поиску номеров методов и свойств объектов 1С.</p> </div> </div> <div class="section" id="id7"> <h1><a name="id7">Вызов методов модулей групповых контекстов</a></h1> <chminfo splitchild="0"> </chminfo> <p><strong>Да!!!
