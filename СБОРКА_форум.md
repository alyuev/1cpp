# Как собрать 1С++ (icpp v3.2.4.3) в рабочий 1cpp.dll — рабочий рецепт

Ниже — только то, что реально понадобилось, чтобы проект **скомпилировался,
слинковался и дал корректный `1cpp.dll`**. Без тупиковых проб. Тулчейн — тот же,
о котором писали на форуме (ICL + MFC42-ABI + STLport + Boost), здесь он доведён
до конца и перечислены конкретные правки в исходниках.

Итог: `119/119 .cpp` компилируются, линковка чистая, `1cpp.dll` (~1.95 МБ,
2 043 904 байта — близко к оригиналу 2 076 672)
экспортирует `DllGetClassObject/DllRegisterServer/DllUnregisterServer/DllCanUnloadNow`
и импортирует рантайм 1С (`BkEnd/Seven/type32/BLang/br32/Editr/Moxel/TxtEdt/Basic/frame`)
плюс `MFC42.DLL + MSVCRT.dll` — то есть ABI `class CString`, совпадающий с либами 1С.

---

## 1. Что установлено и откуда

| Компонент | Версия | Назначение | Откуда |
|---|---|---|---|
| **Intel C++ Compiler** | **11.1.054** Professional, IA-32 | компилятор `icl` (EDG-фронтенд переваривает заголовки MFC42-эпохи и достаточно силён для шаблонов 2010 г.) | установщик Intel |
| **Visual Studio 2005** | — | host-окружение для `iclvars … vs2005` и линкер `link.exe` | дистрибутив VS2005 |
| **Platform SDK for Windows Server 2003 R2** | март 2006 | **MFC 6.0 (`_MFC_VER 0x0600` = `class CString`)** и ATL, которые парсятся современным фронтендом (в отличие от «родных» заголовков VC6) | archive.org, ISO `en_platformsdk_windowsr2_march2006.iso`; админ-распаковка `msiexec /a PSDK-x86.msi` |
| **Visual C++ 6.0** (portable MSVC600) | 6.0 | **CRT (`msvcrt`) и `mfc42.lib`** — правильный ABI для либ 1С | портативный MSVC600 |
| **STLport** | **5.2.1** | STL (нужен `<hash_map>` и `_STLP_*`); **собирается под целевой компилятор** | stlport.org |
| **Boost** | **1.34.1** | header-only (последний Boost, дружелюбный к этой эпохе) | boostorg |

Ключевая идея когерентности: **всё сводится к одному CRT — `msvcrt` (VC6)**.
`mfc42.dll` тянет `msvcrt`; значит и STLport, и рантайм ICL, и приложение должны
идти через `msvcrt` (`/MD`). Иначе — мультирантайм и падения.

## 2. Флаги компиляции (icl, каждый .cpp)

```
/c /Ob1 /O2 -W0 /EHsc /Qms2 /Qvc8 /Zl /MD /Zm800 /Qwd1738,1744 /Qwe1011 /Qinline-max-size:100
```

Дефайны:

```
/D_AFXDLL /DWIN32 /DNDEBUG /D_ANSI /D_WINDOWS /D_USRDLL /D_AFX_DLL
/D_ATL_STATIC_REGISTRY /D_WIN_DLL /D_MBCS
/D_STLP_USE_STATIC_LIB /D_STLP_NEW_PLATFORM_SDK /D_STLP_USING_PLATFORM_SDK_COMPILER
```

INCLUDE (порядок важен — STLport первым): `STLport ; PSDK2003\Include\mfc ;
PSDK2003\Include\atl ; PSDK2003\Include ; VC6\VC98\Include`.

- `/Qvc8` — режим совместимости, под которым разбираются MFC/ATL из PSDK2003.
- `/Zl` — не вшивать имя дефолтной библиотеки (CRT задаём на линковке явно).
- PCH (`/Yc`) **не используем**: у ICL он ложно инстанцирует `FLT_*/LDBL_*` в
  `_limits.h` STLport. Компилируем каждый файл отдельно — это устраняет проблему.

## 3. Настройка STLport (критично для линковки)

STLport надо **собрать под тот же CRT, что и приложение — `/MD`**. Приложение с
`/MD` + `_AFXDLL` получает `_STLP_RUNTIME_DLL`, а с `_STLP_USE_STATIC_LIB` —
признак «статический STLport поверх динамического рантайма» → пространство имён
**`stlpx_std`**. Если статическую либу STLport собрать по умолчанию (`/MT`), она
выйдет в `stlp_std`, и на линковке будет:

```
unresolved external symbol ?_M_deallocate@__node_alloc@stlpx_std@@...
```

Решение — собрать статическую STLport под `/MD`:

```
nmake WITH_DYNAMIC_RTL=1 release-static
```

(в терминах nmake-мейкфайлов STLport 5.2.1 это переводит `OPT_STATIC_REL` на `/MD`
и добавляет `_STLP_USE_STATIC_LIB`). Тогда либа тоже станет `stlpx_std` и совпадёт
с приложением. Плюс native-заголовки C/C++ STLport должен брать из VC6 CRT
(junction/симлинк `stlport\..\include` и `..\crt` → `VC98\Include`).

## 4. Правки в исходниках проекта (то, что реально было не так)

Всего **4 файла**. Это всё, что потребовалось изменить в коде.

### 4.1 `Source/blptr.h` — шаблонный friend
Ядровый умный указатель `CBLPtr<T>` объявлял `template<class Y> friend class CBLPtr;`
и обращался к приватному `p` у ЧУЖОЙ инстанциации (`CBLPtr<U>`) в конструкторах и
`operator=`. Шаблонный friend здесь лишний/проблемный. Заменено на публичный
аксессор:

```cpp
// было:  template<class Y> friend class CBLPtr;
// стало: публичный метод для кросс-инстанциационного доступа
public:
    T* _get_p() const { return p; }
```

и все `lp.p` у чужой инстанциации → `lp._get_p()` (4 места: копи-конструктор,
member-template конструктор, два `operator=`).

### 4.2 `Source/CreateMetaInfoClasses.cpp` — перехват MFC-исключения по значению
MFC-исключения (`CFileException` и др.) **не копируются** и ловятся только по
указателю. Было `catch(CFileException exception)` — перехват по значению:

```cpp
// было:  catch(CFileException exception)
// стало: catch(CFileException* exception)
```

### 4.3 `Source/ODBC/SQLNumeric.cpp` — вызов `malloc` из inline-asm
В `CSQLNumeric::Load` внутри `__asm` стоял `call dword ptr [malloc]`. В VC6 это
работало, потому что `malloc` резолвился в `__imp__malloc` (указатель импорта), и
`[malloc]` давал адрес функции. ICL так не делает — он не разрешает `malloc` в
memory-операнде и генерирует фиктивный внешний символ (на линковке: `unresolved
external symbol ?e`). Заводим настоящую C-переменную-указатель и зовём через неё:

```cpp
void* (__cdecl *pMalloc)(size_t) = malloc;   // объявить перед __asm-блоком
...
//  call    dword ptr [malloc]   // было
    call    dword ptr [pMalloc]  // стало
```

### 4.4 `Source/icl_compat.cpp` — недостающий `operator delete[]` (новый файл)
CRT VC6 (`msvcrt.lib`) не предоставляет массивный `operator delete[](void*)`.
Добавляем тонкую обёртку к скалярному:

```cpp
#include <new>
void operator delete[](void* p) throw() { ::operator delete(p); }
```

Файл добавляем в список компилируемых .cpp.

## 5. Линковка

**Линкуем `link.exe` из VS2005, НЕ `xilink`.** `xilink` при своём IPO-проходе
теряет значение опции `/DEF:` (получается `LNK1146: no argument specified with
option '/DEF:'`). Объектники — обычный COFF (без `/Qipo`), поэтому `link.exe`
подходит; рантайм-либы Intel добавляем руками.

Опции линковки:

```
/dll /MACHINE:IX86 /DEF:Source\1CPP.DEF /BASE:0x24000000
/IGNORE:4199 /NODEFAULTLIB:atlthunk.lib /NODEFAULTLIB:stlport_statix.5.1.lib /FORCE:MULTIPLE
/OUT:1CPP.dll  <все .obj>  1CPP.res
```

LIBPATH: `STLport\lib ; VC6\VC98\MFC\Lib ; VC6\VC98\Lib ; PSDK2003\Lib ;
Source\LIBS ; Source\Forwarder\Release ; Intel\...\lib\ia32`.

Библиотеки. **Порядок КРИТИЧЕН: `mfcs42.lib` должен идти ПЕРЕД `msvcrt.lib`**
(см. п. 5.1 ниже):

```
# ПЕРВЫМИ — статические glue-либы MFC (иначе DllMain не тот, см. 5.1):
mfcs42.lib mfc42.lib
# import-либы 1С (Source\LIBS) — перечислять ЯВНО:
bkend.lib seven.lib type32.lib blang.lib br32.lib dbeng32.lib editr.lib
moxel.lib mxl2xl.lib rgproc.lib txtedt.lib userdef.lib BASIC.lib FRAME.lib
# мост wchar_t (готовый Intel-билд Source\Forwarder\Release\forwarder.lib):
forwarder.lib
# рантайм Intel — СТАТИЧЕСКИ (чтобы не тянуть libmmd.dll, см. п.6):
libmmt.lib libirc.lib libdecimal.lib svml_disp.lib
# CRT/система (msvcrt ПОСЛЕ mfcs42!):
msvcrt.lib kernel32.lib user32.lib gdi32.lib winspool.lib comdlg32.lib
advapi32.lib shell32.lib ole32.lib oleaut32.lib uuid.lib odbc32.lib odbccp32.lib
Rpcrt4.lib winmm.lib version.lib msimg32.lib shlwapi.lib
```

Тонкости, без которых линковка не идёт:

- **`forwarder.lib`** даёт символы пространства `N::` (мост `wchar_t`/`CString` к
  MFC42). Берём готовый Intel-билд из `Source\Forwarder\Release`. Он собран под
  STLport **5.1**, поэтому в нём зашита auto-link прагма `stlport_statix.5.1.lib` —
  гасим её через `/NODEFAULTLIB:stlport_statix.5.1.lib` (символы даёт наш
  `stlpx_std` из STLport 5.2).
- **`atlthunk.lib`** гасим `/NODEFAULTLIB:atlthunk.lib` (прагма из `atlbase.h`
  PSDK2003; сам линк её не требует).
- **Ресурсы**: `rc /l 0x419 /d NDEBUG /d _AFXDLL /I PSDK2003\Include\mfc /I Source
  /fo 1CPP.res Source\1CPP.rc` (без `/nologo` — иначе `RC1106`). GUID-файл
  `1CPP_i.c` компилируем и добавляем в линк.

### 5.1 САМОЕ ВАЖНОЕ: DllMain из MFC, а не из msvcrt

Симптом: линковка проходит, но **1С не запускается с компонентой** — при
`ЗагрузитьВнешнююКомпоненту` падает с access violation в `frame.dll`. Причина —
предупреждение линковки, на которое легко не обратить внимания:

```
mfcs42.lib(dllmodul.obj): warning LNK4006: _DllMain@12 already defined in
                          msvcrt.lib(dllmain.obj); second definition ignored
```

`msvcrt.lib` содержит **пустой** `DllMain`, а `mfcs42.lib` — **настоящий**, который
инициализирует MFC (`AfxWinInit`, состояние модуля, конструирование `theApp`).
Если первым слинкуется msvcrt — MFC НЕ инициализируется, и при загрузке в 1С
компонента рушится. Лечится порядком: **`mfcs42.lib` (и `mfc42.lib`) — ПЕРВЫМИ
в списке, до `msvcrt.lib`**. Тогда LNK4006 исчезает, `theApp` конструируется,
компонента грузится и работает. Это был ключевой фикс: до него DLL линковалась и
даже экспортировала всё правильно, но 1С с ней не стартовала.

Проверять надёжно только **внутри 1С**: 1С++ нельзя загрузить голым `LoadLibrary`
вне 1cv7 (зависимость `Basic.dll` сама инициализируется только в хосте) — так
падает и оригинал; это НЕ признак поломки.

## 6. Самодостаточность (без libmmd.dll)

Оригинальный `1cpp.dll` ни от чего лишнего не зависит — только MFC42/MSVCRT/рантайм
1С. Чтобы так же: рантайм Intel линкуем **статически** — `libmmt.lib` (статическая
math вместо import-либы `libmmd.lib`) + `libirc.lib` + `libdecimal.lib` +
`svml_disp.lib` (статический SVML вместо `svml_dispmd.lib`). Тогда `dumpbin
/DEPENDENTS` показывает только `MFC42.DLL`, `MSVCRT.dll`, рантайм 1С и системные —
**никакой `libmmd.dll`**. (Если по ошибке взять `libmmd.lib`/`svml_dispmd.lib` —
получите зависимость от `libmmd.dll`, которой в оригинале нет.)

## Короткое резюме

1. **Тулчейн:** ICL 11.1.054 + `/Qvc8` + MFC 6.0 из **Platform SDK 2003 R2** +
   CRT/mfc42 из **VC6** + **STLport 5.2.1** + **Boost 1.34.1**. Единый CRT = `msvcrt`.
2. **STLport** — обязательно собрать статическую под `/MD` (`WITH_DYNAMIC_RTL`),
   иначе несовпадение пространств имён `stlp_std` vs `stlpx_std`.
3. **4 правки в коде:** `blptr.h` (убрать шаблонный friend → аксессор `_get_p()`),
   `CreateMetaInfoClasses.cpp` (ловить MFC-исключение по указателю),
   `SQLNumeric.cpp` (вызов `malloc` из asm — через C-указатель),
   `icl_compat.cpp` (доопределить `operator delete[]`).
4. **Линковать `link.exe`, не `xilink`**; явно перечислить import-либы 1С,
   `forwarder.lib` и рантайм Intel; погасить прагмы `atlthunk.lib` и
   `stlport_statix.5.1.lib`.
5. **`mfcs42.lib` ПЕРЕД `msvcrt.lib`** — иначе побеждает пустой `DllMain`, MFC не
   инициализируется, и 1С падает при загрузке компоненты (ключевой фикс, п. 5.1).
6. Рантайм Intel — **статически** (`libmmt.lib`, `svml_disp.lib`), тогда `1cpp.dll`
   самодостаточна, как оригинал: без `libmmd.dll`.
7. Проверять **только внутри 1С** (голый `LoadLibrary` вне 1cv7 не годится — падает
   и оригинал). Готовая пустая тест-база с авто-загрузкой компоненты — в `TestBase\`
   (запуск `run_base.ps1`, результат в `test_log.txt`).
